## 🧠 Parameter Science: Macro & Deep-Dive Mechanics

### 1. Context Window (`num_ctx`)
*   **Macro Explanation**: The "brain capacity" or working memory size of the LLM for a single request. It dictates the combined limit of the input prompt (System prompt + Criteria + Transcript) and the output response.
*   **Deep-Dive Mechanics**:
    *   **VRAM Allocation**: In high-throughput servers like `vLLM`, memory is allocated statically to store the **KV (Key-Value) Cache** for every active sequence. Increasing `num_ctx` linearly scales the VRAM required for KV cache storage.
    *   **Llama 3.1 Capabilities**: The loaded `Meta-Llama-3.1-8B-Instruct-AWQ` model natively supports a **131,072 token context window**! However, local GPUs are hardware-constrained. Setting this too high without matching physical VRAM will crash the container.
    *   **Optimal Setting**: Use `8,192` for short calls to preserve memory bandwidth, and scale up to `16,384` or `32,768` for long calls.

### 2. Max Predict Tokens (`num_predict` / `max_tokens`)
*   **Macro Explanation**: The maximum size of the output response generated by the LLM. 
*   **Deep-Dive Mechanics**:
    *   **The Validation Trap**: When an API request is made, `vLLM` reserves a hard slice of memory equal to `max_tokens`. It calculates: $\text{Prompt Tokens} + \text{Max Tokens} \le \text{Max Model Len}$.
    *   If this sum exceeds the context window, it immediately rejects the call with a `400 Bad Request` *even if the actual answer would have only taken 100 tokens*.
    *   **Optimal Setting**: For longer calls, restrict `num_predict` to `1,024` or `1,536`. Since the model only outputs a JSON array of criteria scores and justifications, it rarely needs more than 500 tokens. Restricting this leaves maximum room for the incoming transcript.

### 3. Temperature & Sampling (`top_p`, `top_k`)
*   **Macro Explanation**: Dictates how creative, random, or strict the model's choices are.
*   **Deep-Dive Mechanics**:
    *   **Temperature (`0.0`)**: Forces the model to select the absolute highest-probability token at every step (greedy decoding). This is **critical** for compliance audits, ensuring that if you re-run the same transcript, you get the exact same deterministic score and reasoning.
    *   **Top P (`0.90`) / Top K (`40`)**: Acts as a cumulative probability filter before decoding. Leaving this at `0.90` allows the model minor semantic flexibility to formulate natural justification sentences without compromising determinism.

## 🛑 Critical Diagnosis: Why `aiaudit-vllm` Exits with Code 137

If the `aiaudit-vllm` container exits with **code 137**, Docker has terminated the server because it exceeded the **system RAM limit** (Out Of Memory) of the host or WSL2 virtual machine.

### ⚠️ Root Cause: WSL2 Memory Starvation
1. **Model Weights Loading**: When `vLLM` initializes, it first loads the 8B model weights into **system RAM** before copying them over to GPU VRAM. During weight initialization and CUDA graph compilation, system RAM usage spikes heavily (typically requiring **10GB to 14GB of RAM** temporarily).
2. **WSL2 Limits**: On Windows, WSL2 restricts virtual machine RAM to **50% of the host PC's RAM** by default. For example, on a 16GB RAM PC, WSL2 is strictly capped at 8GB.
3. **The Kill**: When the combined footprint of PostgreSQL, Redis, Celery workers, and the vLLM engine exceeds this cap, the WSL2 OOM (Out Of Memory) Killer terminates the `vllm` container instantly, producing exit code `137`.

### 🛠️ Step-by-Step Resolution Plan

#### Step A: Configure `.wslconfig` (Permanent Fix)
To prevent the host OS from killing your GPU server, you should allocate more system RAM to WSL2:
1. Open your Windows File Explorer and navigate to your user home directory (e.g. `C:\Users\<YourUsername>`).
2. Check if a file named **`.wslconfig`** exists. If not, create a new text file named exactly `.wslconfig`.
3. Add the following configuration block (recommending at least 16GB if you have a 32GB RAM PC, or adjusting to 75% of your host RAM):
   ```ini
   [wsl2]
   memory=16GB       # Allocate 16GB of system RAM to WSL2
   processors=4      # Optional: restrict CPU cores to prevent lockups
   ```
4. Save the file.
5. Restart WSL2 by opening PowerShell on your Windows Host and running:
   ```powershell
   wsl --shutdown
   ```
6. Restart your Docker Desktop. When Docker boots up, it will mount into the newly expanded memory envelope.

#### Step B: Optimize vLLM Startup Arguments
You can reduce vLLM's memory allocation at startup. Open your `docker-compose.yml` file and review the `vllm` command block:
*   Ensure `--gpu-memory-utilization` is set to a reasonable margin (like `0.80` or `0.85`) to leave GPU resources available.
*   Keep `--max-model-len` at `8192` or `16384` to prevent oversized initial CUDA graph generation.

#### Step C: Align the STT Model Size
For call centers regularly auditing calls longer than 10 minutes, **avoid heavy Whisper sizes**.
*   Under **Transcription (STT) Engine**, change the **Whisper Model Size** to **`Tiny`** or **`Small`**.
*   *Why?* The `tiny` Whisper model consumes negligible system RAM and VRAM (~70MB), completely avoiding memory contention with vLLM.
